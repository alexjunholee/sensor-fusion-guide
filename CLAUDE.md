# sensor-fusion-guide

센서 퓨전 심화 가이드. robotics-practice보다 깊은 수준의 레퍼런스.

## 구조

- `guide.html` — 메인 가이드 (단일 HTML)
- `chapter_*.md` — 각 챕터 원고 (독립 Markdown 파일)
- `interactive/` — 인터랙티브 시각화/데모 모음 (HTML/JS)
- `scripts/` — 빌드/유틸리티 스크립트
- `outlines/` — 구상 단계의 아웃라인 파일
- `writing-status.json` — 콘텐츠 작성 파이프라인 현황

## 작업 규칙

- HTML/JS 기반 프로젝트. 빌드 도구 없이 브라우저에서 직접 실행.
- 인터랙티브 데모 수정 시 `interactive/` 디렉토리 내에서 작업.
- git push 전 반드시 확인 요청.

## 대상 독자

- 로보틱스 입문자이지만 선형대수, 확률론 기초는 아는 수준
- robotics-practice guide보다 깊고 실전적인 내용
- 센서 퓨전을 실제로 구현/연구하는 사람을 위한 레퍼런스

## robotics-practice와의 차별

- robotics-practice는 Spatial AI 전반의 입문 브레드스
- 이 가이드는 센서 퓨전/로컬라이제이션/리트리벌에 집중한 심화 레퍼런스
- 겹치는 기초 내용은 robotics-practice 참조로 대체, 여기서는 깊이 있게 다룸

## 글쓰기 파이프라인 (4단계)

이 프로젝트의 콘텐츠는 반드시 4단계 파이프라인을 따라 작성한다.
**단계를 절대 합치지 않는다.** 각 단계는 별도의 세션/요청에서 수행한다.

### 작업 시작 프로토콜

1. `writing-status.json`을 읽는다.
2. 사용자가 지정한 콘텐츠 유닛의 현재 stage를 확인한다.
3. **해당 stage의 체크리스트만 수행한다.** 다른 stage의 작업은 하지 않는다.
4. 작업 완료 후 `writing-status.json`의 stage를 다음 단계로 업데이트한다.

사용자가 stage를 지정하지 않으면 `writing-status.json`의 현재 stage를 보고하고 확인을 요청한다.

---

### Stage 1: 구상 (Prewriting) — 사용자 주도

**목표**: 사용자의 아이디어를 구조화된 아웃라인으로 정리.

체크리스트:
- [ ] 주제, 범위, 대상 독자 확정
- [ ] 섹션/하위섹션 구조 확정
- [ ] 각 섹션의 핵심 포인트(key points) 3~5개 나열
- [ ] 인터랙티브 데모 포함 계획 명시
- [ ] 톤/난이도 지침 (예: 신입생 대상, 직관→수식→코드)
- [ ] `outlines/{unit_name}.md`에 저장
- [ ] `writing-status.json`에 유닛 등록, stage → `2-drafting`

**하지 않는 것**: 본문 작성, 문장 다듬기, 수식 정리

---

### Stage 2: 초고 (Drafting) — Claude

**목표**: 아웃라인의 모든 포인트를 빠짐없이 본문으로 전개.
**핵심**: 완벽함보다 완성. 아웃라인에 있으면 불완전해도 지금 쓴다.

체크리스트:
- [ ] `outlines/`에서 아웃라인 파일 읽기
- [ ] 아웃라인의 모든 key point가 본문에 반영 (누락 금지)
- [ ] 각 섹션: 직관 → 수식 → 코드/예제 흐름
- [ ] 코드 예시가 필요한 곳에 코드 블록 포함 (완벽하지 않아도 됨)
- [ ] 인터랙티브 데모 위치를 `<!-- DEMO: filename.html -->` 주석으로 표시
- [ ] `writing-status.json` → `3-revising`, `drafting_date` 기록

**절대 하지 않는 것**:
- 문장을 다듬거나 우아하게 만들기
- 표기법 통일
- 내용 삭제 또는 축약
- "나중에 보강" TODO 남기기

---

### Stage 3: 수정 (Revising) — Claude

**목표**: 기술적 정확성과 논리적 흐름 검증. 내용(substance)만 본다.

체크리스트:
- [ ] **수식 정확성**: 행렬 차원, 부호, 인덱스, 자코비안, 확률 분포 적용
- [ ] **개념 정확성**: 알고리즘 설명, 센서 모델, 필터 유도 과정
- [ ] **논리적 흐름**: 정의 없이 등장하는 용어 없는가, 난이도 순서가 맞는가
- [ ] **코드 정확성**: import 누락, 변수명 일관성, numpy/scipy 호출 올바른가
- [ ] **데모 연동**: 데모에서 사용하는 개념이 본문에서 설명되었는가
- [ ] **구조 조정**: 섹션 순서 변경, 분리/병합 필요한 곳
- [ ] `writing-status.json` → `4-editing`, `revising_date` 기록

**하지 않는 것**: 문체 개선, 오타 수정, KaTeX 표기 통일, HTML 포매팅 정리

---

### Stage 3b: 흐름 (Coherence) — Claude

**목표**: 챕터·섹션 간의 내러티브 연결성을 확보한다. 독자가 "왜 지금 이 내용을 읽는가"를 항상 알 수 있도록 한다.
**핵심**: 기술적 내용 자체는 건드리지 않는다. 추가하는 것은 오직 **도입·전환·마무리 문단**이다.

작업 단위: 한 번의 요청에 **3~4챕터씩** 진행한다 (예: Ch.1-3, Ch.4-6, ...).

체크리스트 — 챕터 단위:
- [ ] **챕터 도입(opening)**: 이 챕터가 전체 가이드에서 어디에 위치하는지, 앞 챕터와 어떻게 이어지는지 1-2문단으로 설명. "Ch.N에서 …을 다루었다. 그러나 …이 빠져 있다. 이 챕터에서는 …"
- [ ] **챕터 마무리(closing)**: 이 챕터의 핵심 takeaway 2-3줄 요약 + 다음 챕터 예고. "이제 …를 갖추었으니, Ch.N+1에서는 이를 활용하여 …"
- [ ] **섹션 간 전환(transition)**: 각 `##` 섹션이 시작될 때, 이전 섹션과의 논리적 연결이 있는가? 없으면 1-2문장 추가. "앞 절에서 …의 원리를 보았다. 이제 이를 …에 적용한다."
- [ ] **갑작스러운 시작 점검**: 섹션 첫 문장이 정의나 수식으로 돌입하는 경우, "왜 이것을 다루는가"를 한 문장이라도 앞에 추가
- [ ] **갑작스러운 끝 점검**: 코드 블록이나 수식으로 끝나는 섹션에, "이로써 …를 확인했다" 식의 마무리 1문장 추가
- [ ] **챕터 간 교차 참조**: 다른 챕터의 개념을 참조할 때 "(Ch.N 참조)" 형태로 명시. 독자가 점프할 수 있도록.

점검 원칙:
- **추가만 한다**: 기존 기술 내용을 삭제하거나 순서를 바꾸지 않는다
- **간결하게**: 전환 문단은 1-3문장. 장황한 요약은 쓰지 않는다
- **독자 관점**: "이 섹션을 처음 만난 독자가 왜 이걸 읽어야 하는지 아는가?"로 판단
- **notes에 기록**: 챕터별로 수정 건수와 유형 (도입 추가 N건, 전환 추가 N건 등)을 writing-status.json의 notes에 기록

`writing-status.json` → `4-editing`, `coherence_date` 기록

**하지 않는 것**: 기술 내용 수정, 수식/코드 변경, 표기법 통일, 오타 수정

---

### Stage 4: 교정 (Editing) — Claude (검수자)

**목표**: 표면 품질을 프로젝트 전체 수준으로 끌어올린다. 내용은 건드리지 않는다.

체크리스트:
- [ ] **수학 표기 통일**: 벡터(`\mathbf{v}`), 행렬(`\mathbf{M}`), 좌표계(`{}^{W}T_{B}`) — 기존 방식 따름
- [ ] **코드 스타일 통일**: 포매팅, 주석 언어, 변수명 컨벤션(snake_case)
- [ ] **용어 통일**: 한영 혼용 패턴 (예: "기구학(Kinematics)"), 동일 개념 동일 번역
- [ ] **HTML/Markdown 포매팅**: 헤딩 레벨, blockquote 스타일, 데모 임베드 방식 기존과 일치
- [ ] **오타/문법**: 맞춤법, 조사 오류, 영어 스펠링
- [ ] **읽기 흐름**: 문단 전환, 문장 길이
- [ ] `writing-status.json` → `done`, `editing_date` 기록
